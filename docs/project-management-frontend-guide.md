# 项目管理模块前端接入规范

## 文档目标

本文档用于沉淀项目管理模块在前端项目中的长期接入约定，包括页面结构、菜单组织、权限对齐和交互规则，不记录单次接入过程。

## 一、模块范围

当前项目管理模块前端范围包括：

- 业务线
- 需求管理
- 缺陷管理
- 业务线统计

对应页面文件包括：

- `src/pages/Projects.jsx`
- `src/pages/Requirements.jsx`
- `src/pages/Bugs.jsx`
- `src/pages/ProjectStats.jsx`
- `src/pages/ProjectManagement.css`

## 二、接入结构约定

### 1. API 层

项目管理模块 API 按领域拆分：

- `src/api/projects.js`
- `src/api/requirements.js`
- `src/api/bugs.js`
- `src/api/projectStats.js`

规则如下：

- 每个领域对象独立维护 API 封装
- 列表、详情、创建、编辑、删除、状态动作尽量保持清晰分层
- 不把所有项目管理请求堆到一个 API 文件中

### 2. 页面层

页面按业务领域拆分，不共用超大页面。

推荐模式：

- 列表页
- 筛选区
- 表格区
- 弹窗或抽屉表单
- 详情补充区

当前项目优先采用“列表 + 筛选 + 弹窗表单”的 MVP 形式。

## 三、菜单与路由约定

项目管理模块在前端侧以独立菜单分组呈现，包含：

- 业务线
- 需求管理
- 缺陷管理
- 业务线统计

相关配置统一维护在：

- `src/config/route.config.js`
- `src/config/menu.config.js`

规则如下：

- 菜单名称、页面标题和副标题保持一致的语义层级
- 新增页面优先进入项目管理分组，不随意散落到其他菜单下
- 路由元信息与页面权限要求保持同步

## 四、权限对齐规则

前端继续复用现有权限判断体系，不单独建设新的权限方案。

当前模块核心权限包括：

- `project.view`
- `requirement.view`
- `bug.view`
- `project.stats.view`

后续如新增动作权限，应优先与后端权限编码保持一一对应，避免前后端出现不同命名体系。

## 五、页面交互规则

### 1. 通用风格

- 保持现有 `antd` 页面风格
- 保持当前后台的左侧菜单和顶部页头结构
- 保持现有请求封装和消息提示方式

### 2. 业务线页

- 以业务线列表为主入口
- 支持业务线详情、成员管理、活动日志
- 详情信息优先在当前页面延展展示，不增加复杂跳转层级

### 3. 需求与缺陷页面

- 以列表和筛选为主
- 高价值字段如优先级、状态、阶段、负责人、工时应直接可见
- 高频操作优先使用弹窗完成，减少页面切换成本

### 4. 统计页面

- 保持总览、按业务线、按成员三类视角
- 统计口径必须与后端返回一致
- 若后续增加图表，图表只负责展示，不重新定义统计逻辑

## 六、数据展示规则

- 前端展示文案可以中文化
- 当前项目管理模块在前端展示层将“项目”文案统一映射为“业务线”
- 提交给后端的状态值、阶段值和枚举值必须保持约定编码
- 负责人展示优先使用 `real_name`，为空时回退 `username`
- 所有工时统一按小时显示，人天显示需与后端口径一致

## 七、样式维护规则

- 项目管理模块的页面样式尽量集中在 `src/pages/ProjectManagement.css`
- 仅当页面差异明显时，才拆分独立样式文件
- 保持列表页、筛选区、统计卡片和详情区的视觉一致性

## 八、后续扩展原则

后续如继续扩展项目管理模块前端能力，优先按以下方向演进：

- 需求与 Bug 的快捷状态流转
- 项目详情联动视图
- 统计图表展示
- 看板式视图

扩展时优先复用现有模块分层，不直接打破当前页面和 API 组织方式。

## 九、文档记录规则

后续与项目管理模块前端相关的文档，只记录以下内容：

- 页面结构约定
- 权限对齐规则
- API 分层规则
- 样式和交互规则

不记录以下内容：

- 某次接入新增了哪些文件
- 某次构建或 lint 执行结果
- 某次联调的逐步过程
